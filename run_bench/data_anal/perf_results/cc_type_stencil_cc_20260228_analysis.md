# 为什么 grid size 很大时不同 cc_type 性能差异体现不出来

## 数据现象总结

| grid_size | 典型 cc_type 差异 (vs nocc) | 说明 |
|-----------|-----------------------------|------|
| **1000**  | 高进程数时极大：P=36 约 **+42% ~ +49%**；P=4 约 ±0.5% | 小网格 + 多进程时差异非常明显 |
| **2000**  | P=36 约 +4% ~ +7%；其他 P 约 ±1% ~ ±2% | 中等差异 |
| **4000**  | 多数在 **±1% 以内**，P=25/36 尤其小 | 差异明显收窄 |
| **8000**  | 几乎全部在 **±1% 以内**，多数 ±0.5% | 不同 cc_type 几乎无差别 |

结论：**grid size 越大，不同 cc_type（nocc / clflush / clwb 等）之间的性能差异越小；grid size 小且进程数多时，差异最大。**

---

## 原因分析

### 1. 计算 vs 缓存一致性开销的比例（主因）

- **cc_type 的差异**主要来自：不同 cache coherence 策略（nocc / clflush / clwb 等）在 **RMA 同步、写回、冲刷缓存** 上的开销不同。
- 这些开销可以近似看成**与“需要做一致性操作的内存访问次数”相关**，和总数据量、通信次数有关，但**不随单进程计算量成比例增加**。

- **大 grid size**：
  - 每个进程分到的子域大，** interior 计算量**（纯算、不直接涉及 halo 同步）远多于 halo 通信与写回。
  - 总时间中，**计算占主导**，一致性操作占的比例小 → 换一种 cc_type 只改动了总时间中很小一块 → **性能差异被“淹没”**，体现不出来。

- **小 grid size + 多进程**：
  - 每个进程子域很小，** halo 通信 / 写回 / 同步** 在总时间中占比高。
  - 一致性策略（nocc 不刷 vs clflush/clwb 刷）直接改变这部分时间 → **性能差异大**，所以能明显看出不同 cc_type。

### 2. 表面积 / 体积比（Stencil 特性）

- Stencil 中：
  - **“体积”** ≈ 每个进程 interior 格点数（与计算量成正比）；
  - **“表面积”** ≈ halo 交换、RMA put/get 涉及的数据量（与一致性、同步次数相关）。
- 总格点增加时，体积增长比表面积快（例如 3D 时体积 ∝ n³，表面积 ∝ n²），所以：
  - **grid 越大** → 计算主导，通信/同步占比下降 → **cc_type 影响变小**；
  - **grid 小、进程多** → 每个进程表面积/体积比大 → 同步与一致性开销占比高 → **cc_type 影响大**。

### 3. 一致性开销被“摊薄”

- clflush / clwb 等可以粗略看成**按 cache line 或按写操作**产生的固定开销。
- **大 grid**：每个进程做更多次计算、更多次内存访问，但**需要做显式一致性的写/同步次数**相对增加得少 → 单次一致性成本被大量计算摊薄，**占总时间比例下降** → 换 cc_type 带来的绝对时间差在总时间里占比小，**相对差异就体现不出来**。

### 4. 内存带宽 vs 一致性延迟

- **大 grid**：更容易把内存带宽吃满，整体处于 **memory-bandwidth bound**，所有 cc_type 都受同一瓶颈限制，绝对时间接近，**相对差异小**。
- **小 grid + 多进程**：未必打满带宽，此时 **延迟**（同步、写回、一致性指令）更敏感，nocc 与各种 clflush/clwb 策略在延迟上的差别就会表现为明显的**性能差异**。

---

## 小结

| 因素 | 大 grid size | 小 grid size（尤其多进程） |
|------|-------------|----------------------------|
| 计算占比 | 高，主导总时间 | 低，同步/通信占比高 |
| 一致性开销占比 | 低，被摊薄 | 高，直接决定差异 |
| 瓶颈 | 多为带宽 | 多为延迟/同步 |
| cc_type 差异 | **体现不出来**（±1% 内） | **非常明显**（可达数十 %） |

因此，**当 grid size 很大时，不同 cc_type 的性能差异体现不出来**，主要是因为：总时间主要由计算和内存带宽决定，而 cache coherence 策略带来的额外开销在总时间中占比很小，被“淹没”了。要观察 cc_type 的影响，应优先看 **小 grid、多进程** 的配置（例如 grid=1000、P=36）。
